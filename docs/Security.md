# セキュリティ設計

## 1. 目的

本セキュリティ設計書は、アプリケーションを安全に運用するために以下を定義する。

- 認証・認可の方式と責任範囲
- 通信・データ保護の方針
- Web脆弱性対策（OWASP観点の最低限）
- 外部サービス利用（Firebase/Stripe/OpenAI 等）の安全な扱い
- ログ・運用における機密情報の扱い

---

## 2. セキュリティ要件（MVP基準）

### 2.1 達成すべき最低条件

- **認証必須**：ユーザー識別が必要な機能（日記作成/編集/削除、マイページ、課金機能）はログイン必須
- **認可必須**：ユーザーの作成物（日記、訪問記録）は本人以外が操作できない
- **HTTPS必須**：本番はTLSで暗号化通信のみ
- **秘密情報は環境変数管理**：APIキー、DBパスワード等をソースに含めない
- **ログに機密情報を出さない**：トークン、キー、日記本文など

---

## 3. 脅威モデル（前提）

MVPで特に守るべきリスクは以下。

- 不正ログイン / なりすまし
- 他人の日記データ閲覧・改ざん（IDOR）
- トークン漏洩（ログ、ブラウザ、通信）
- 画像アップロードを悪用した攻撃（巨大ファイル、拡張子偽装、公開URL悪用）
- 決済偽装（Stripeの成功/失敗の信頼性）
- AI機能の乱用（大量リクエスト、プロンプトに機密混入）

---

## 4. 認証（Authentication）設計

### 4.1 方式

- **Firebase Authentication** を利用
- フロントでログイン → Firebase IDトークン取得 → APIに送信

### 4.2 トークン取り扱い

- APIへは `Authorization: Bearer <id_token>` で送る
- API側は毎回トークン検証し、`uid` を確定する
- **禁止**：トークンをログ出力しない

### 4.3 セッション方針（MVP）

- フロントはFirebase SDKに任せる（トークン更新はSDKが実施）
- APIは「検証されたトークン」以外は信用しない（クライアント送信のuser_idを信用しない）

---

## 5. 認可（Authorization）設計

### 5.1 基本原則

- **サーバー側で必ず認可判定**
    
    → 「このユーザーがこの日記を操作できるか」をAPIで確認する
    

### 5.2 ルール（MVP）

| 対象 | 操作 | ルール |
| --- | --- | --- |
| diary | create | ログインユーザーのみ |
| diary | read | 公開/非公開仕様に従う（MVPは本人のみでもOK） |
| diary | update/delete | **ownerのみ**（`diary.user_id == current_user_id`） |
| visit | create/update/delete | ownerのみ |
| payment | create | ログインユーザーのみ |
| admin操作 | 全般 | MVPでは未実装 or 管理者のみ |

### 5.3 IDOR対策（重要）

- URLに `/diaries/{id}` がある場合でも、**id指定だけでアクセス許可しない**
- 必ず `is_owner` をDBで判定して拒否（403）

---

## 6. 通信・暗号化（Transport Security）

### 6.1 HTTPS

- 本番はHTTPSのみ（HTTPはリダイレクト or 拒否）
- 外部サービス連携（Stripe/OpenAI/Maps）もHTTPSのみ

### 6.2 CORS

- CORS許可は **フロントのドメインに限定**
- 許可は禁止（開発環境のみ例外で限定許可）

---

## 7. 入力バリデーション・Web脆弱性対策（OWASP最低限）

### 7.1 SQL Injection

- SQLAlchemyのORM利用＋パラメータバインド（生SQLを極力避ける）
- 生SQLが必要な場合もバインド変数を使用

### 7.2 XSS

- ユーザー入力（日記本文等）は **HTMLとして描画しない**
- `dangerouslySetInnerHTML` 原則禁止
- 表示はテキストとしてエスケープ

### 7.3 CSRF

- Bearerトークン方式であればCSRFリスクは相対的に下がる
- ただしCookie運用をする場合はSameSite/Lax or Strict等を検討（MVPはトークンヘッダ推奨）

### 7.4 不正リクエスト/DoS（最小対策）

- AI推薦など高コストAPIは **レート制限** を検討（IP/ユーザー単位）
- 失敗リトライの上限（バックエンド側で制限）

---

## 8. 画像アップロード（Storage）セキュリティ

### 8.1 アクセス制御

- Firebase Storageのセキュリティルールで制御（認証必須 or 公開範囲を明確化）
- 公開する場合でも、推測しにくいパス設計（例：`users/{uid}/...`）

### 8.2 アップロード制約

- 許可拡張子：jpg/png/webp（必要に応じて）
- ファイルサイズ上限：例 5MB
- Content-Type検証（拡張子偽装対策）
- 画像処理（将来）：サムネ生成やメタ情報削除（EXIF）

### 8.3 URLの扱い

- 画像URLはDBに保存するが、外部に不用意に露出させない
- 署名付きURLが必要な場合は有効期限を設定（MVPは運用方針で決める）

---

## 9. 決済（Stripe）セキュリティ

### 9.1 基本方針

- 決済結果は **クライアントの結果だけを信用しない**
- 可能ならStripeのWebhookイベントで「成功」を確定する（MVPで難しければ“サーバーで検証”だけでも）

### 9.2 秘密情報の保護

- Stripe secret key / webhook secret は環境変数
- ログ出力禁止

### 9.3 重要ログ（監査ログ）

- 決済開始/成功/失敗は監査ログとして残す（ユーザーID、イベントID、結果）

---

## 10. AI機能（OpenAI）セキュリティ

### 10.1 送信データの最小化

- プロンプトに **個人情報（日記全文、メール、位置情報の詳細）を送らない**
- 送るなら要約・匿名化（例：「20代女性」などは避け、単に「ユーザー」程度）

### 10.2 機密情報の混入防止

- APIキーは必ずサーバー側で保持（フロントに出さない）
- エラー時にプロンプト全文をログに出さない（要約に留める）

### 10.3 乱用対策

- 連続実行の制限（レート制限）
- タイムアウトと再試行回数制限

---

## 11. 秘密情報管理（Secrets Management）

### 11.1 対象

- DB接続情報
- Firebase設定（特に秘密扱いの値）
- Stripe secret / webhook secret
- OpenAI API key
- Google Maps API key

### 11.2 ルール

- `.env` は本番/開発で分離
- コードベースに埋め込み禁止
- 誤コミット防止（gitignore/secret scanの導入は将来検討）

---

## 12. ログ・監査（Security Logging）

### 12.1 ログ出力禁止

- トークン、パスワード、APIキー、日記本文の全文

### 12.2 監査ログ対象（MVP）

- ログイン成功/失敗（失敗は回数/原因コード程度）
- 日記作成/更新/削除
- 画像アップロード成功/失敗
- 決済イベント

---

## 13. セキュリティテスト（MVP簡易）

### 13.1 重点テスト

- 未ログインで日記作成APIを叩く → 401/403
- 他人の `diary_id` を指定して更新 → 403
- CORSが許可ドメイン以外から通らない
- 画像に巨大ファイルを送る → 拒否
- 決済成功をクライアントだけで偽装できない（サーバー側判定）

---
