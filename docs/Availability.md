# 可用性設計

## 1. 目的

本可用性設計書は、「いさんぽ」アプリケーションを継続的に利用可能にするために、稼働率目標、障害時の影響最小化（劣化運転）、復旧手順、監視方針を定義する。

---

## 2. 可用性目標（SLO）

### 2.1 稼働率

- **月間稼働率：99.5%以上**（MVP目標）

> 99.5%は、月あたりの停止許容がざっくり数時間レベル（約3.6時間程度）になります。
> 
> 
> MVP段階では“高可用性構成”よりも「止まったら戻せる・影響範囲を小さくする」を優先します。
> 

### 2.2 可用性の定義（何ができれば“稼働”か）

MVPの稼働判定は以下を満たす状態とする。

- トップページが表示できる
- 世界遺産一覧・詳細の閲覧ができる
- ログインができる（※Firebase側障害は除外扱いとするか、別枠で扱うかを事前に決める）
- APIが重大エラー（5xx連発）になっていない

---

## 3. システム構成と単一障害点（SPOF）

### 3.1 構成要素（対象）

- フロント：Next.js
- API：FastAPI
- DB：PostgreSQL（Supabase）
- 外部：Firebase Auth / Firebase Storage / Stripe / Google Maps / OpenAI

### 3.2 単一障害点の整理（MVPの現実）

- DB障害 → **最も影響が大きい**（閲覧/投稿が止まる）
- API障害 → フロントが表示できても機能が止まる
- Firebase Auth障害 → ログインが止まる（サービス利用不可寄り）
- Storage障害 → 画像が出ない（機能は部分的に継続可能）
- Stripe障害 → 新規決済不可（既存閲覧は継続可能にする）
- Maps/OpenAI障害 → 一部機能停止（劣化運転可能）

---

## 4. 劣化運転（Graceful Degradation）設計

可用性の要は「全部止めない」ことなので、外部依存ごとに劣化パターンを定義します。

### 4.1 劣化運転ポリシー（一覧）

| 障害対象 | 影響 | 劣化運転方針 |
| --- | --- | --- |
| Firebase Auth | ログイン不能 | 「ログイン機能停止中」表示＋再試行誘導（可能なら閲覧専用モードへ） |
| Firebase Storage | 画像取得/アップロード不能 | 画像をプレースホルダー表示、投稿は画像なしで継続可（画像は後で追加） |
| Stripe | 決済不能 | 決済導線を停止（ボタン非表示/無効）、既存課金状態の閲覧は継続 |
| Google Maps | 地図表示不可 | 地図ページのみ停止、一覧検索導線は継続 |
| OpenAI | レコメンド生成不可 | AI導線を停止、代替として「人気/おすすめ固定」表示などに切替（MVPなら“停止中表示”でも可） |

※ この表を「UIの表示メッセージ」「ボタン制御」「APIのエラー文言」とセットで実装できると強いです。

## 5. 障害検知（監視）設計

### 5.1 監視レベル（MVP推奨）

MVPでは複雑な監視基盤よりも、まず **死活監視＋重大エラー検知** を優先します。

### (1) 死活監視（外形監視）

- 監視対象
    - フロント：トップページ URL
    - API：`/health`（簡易ヘルスチェック）
- 判定
    - 連続失敗（例：3回）で障害扱い

### (2) API健全性監視（ログ/メトリクス）

- 5xx率が一定以上（例：5分間で 1%超） → 異常
- P95遅延がSLO超過 → 性能劣化（可用性低下の前兆）

### (3) 外部サービス監視

- Stripe / Firebase / OpenAIの失敗率急増をログから把握
- 重要：障害切り分けができるように外部連携ログに `external_service` を必ず入れる

## 6. ヘルスチェック設計

### 6.1 APIヘルスチェック `/health`

最低限以下を返す。

- ステータス：200 OK
- 内容：`{"status":"ok"}` など
- 可能なら DB接続確認を含める（※重くしない）

**注意**

- `/health` は認証不要
- ただし情報を出しすぎない（内部構成・秘密情報は返さない）

## 7. 復旧設計（DR/復旧目標）

### 7.1 復旧目標（MVP）

| 指標 | 目標 |
| --- | --- |
| RPO（許容データ損失） | 24時間以内 |
| RTO（復旧時間） | 4時間以内 |

### 7.2 復旧シナリオ別の方針

### (1) API障害

- 直近リリースが原因ならロールバック
- 設定（環境変数）変更が原因なら設定差分を戻す
- まず復旧 → 後で恒久対応

### (2) DB障害

- バックアップからリストア
- 重大なデータ不整合が疑われる場合は、まず閲覧を止めて整合性を回復

### (3) 外部サービス障害

- サービス復旧を待つ間、劣化運転に切替
- 影響が大きい場合は「障害情報」ページやバナーで告知

---

## 8. 障害時のユーザー向け挙動（UX要件）

### 8.1 エラーメッセージ方針

- 何が起きているか（例：ログインできない、AIが利用できない）
- ユーザーができること（再試行、時間をおく、別導線）
- サポート先（MVPなら“時間をおいて”でもOK）

### 8.2 フォールバック表示

- 画像が取れない → 画像枠は残しプレースホルダー
- 地図が落ちた → 「地図を表示できません。一覧から探してください」
- AIが落ちた → 「現在AIおすすめは停止中です」

- アラート自動化（Slack通知等）
