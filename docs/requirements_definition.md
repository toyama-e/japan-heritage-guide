## 2. プロジェクト概要

### 2.1 システム名

**いさんぽ JAPAN**

### 2.2 システムの目的・背景

日本の世界遺産に興味はあるものの、

- 内容が難しく理解しづらい
- 見どころが分からないまま訪問してしまう

といった理由から、**訪問体験を十分に楽しめていない一般旅行者**を対象に、
以下の体験を提供することを目的とする。

- 世界遺産の価値を「直感的に理解」できる
- 訪問体験を「コレクション感覚」で記録・蓄積できる

## 3. 用語定義

| 用語     | 定義                                  |
| -------- | ------------------------------------- |
| 世界遺産 | UNESCO に登録された文化遺産・自然遺産 |
| 遺産群   | 複数の構成資産からなる 1 つの世界遺産 |
| 構成資産 | 遺産群を構成する個別の史跡・建造物    |
| 達成率   | 登録済み世界遺産に対する訪問割合      |
| バッジ   | 訪問数に応じて付与される称号・実績    |
| MVP      | 最小限の実用機能を備えたプロダクト    |

---

## 4. 業務フロー（As-Is / To-Be）

### 4.1 現状（As-Is）

1. 世界遺産に興味を持つ
2. Google 検索で情報収集
3. 難解な解説文を読む
4. 訪問するが理解が浅い
5. 記録が残らず終了

➡ **理解・満足・継続が生まれにくい状態**

---

### 4.2 新システム導入後（To-Be）

1. アプリで世界遺産一覧を確認
2. 地図で場所・周辺を把握
3. 「見どころ TOP3」で価値を理解
4. 現地で意識的に見学
5. 写真・コメント付きで訪問記録
6. 達成率・バッジで次の行動につながる

➡ **理解 → 体験 → 記録 → 次の行動** が継続する体験を実現

## 6. 非機能要件

| 項目           | 内容                                                             |
| -------------- | ---------------------------------------------------------------- |
| パフォーマンス | 画面表示は 2 秒以内                                              |
| 可用性         | 月間稼働率 99.5%以上                                             |
| セキュリティ   | HTTPS 通信、JWT 認証、OWASP 対策                                 |

| 対応端末       | Chrome / Safari（最新 2 バージョン）                             |
| 保守性         | 世界遺産追加に耐えられる設計                                     |

## 8. データ要件

### ユーザー（Users）

- **user_id（UUID / PK）**
- **firebase_uid（ユニーク）**
  - ※認証は Firebase 前提のため、**password は DB に保持しない**
- nickname
- email_address（ER 上は `e-mail_adress` 表記）
- is_public（ER 上は `is_pubric` 表記）
- created_at
- updated_at
- deleted_at

---

### 世界遺産（WorldHeritage）

- id（PK）
- name（名称）
- type（文化 / 自然）
- latitude / longitude（緯度・経度）
- address（ER 上は `adress` 表記）
- year
- summary
- 見どころ（spot1_title / spot1_detail、spot2_title / spot2_detail、spot3_title / spot3_detail）
- image_url（世界遺産画像 URL）
- badge_image_url（獲得バッジ画像 URL）
- sites（遺産群 / 構成資産などを表す用途）

---

### 訪問記録（visit）

- id（PK）
- user_id（FK → Users.user_id）
- world_heritage_id（FK → WorldHeritage.id）
- visited_from（訪問開始日）
- visited_to（訪問終了日）
- created_at
- deleted_at

---

### 日記（Diaries）

- id（PK）
- user_id（FK → Users.user_id）
- world_heritage_id（FK → WorldHeritage.id）
- visit_id（FK → visit.id）※任意
- visit_day（訪問日）
- title
- text
- image_url（写真 URL）
- like_count（いいねボタン）
- created_at
- updated_at
- deleted_at

## 9. 外部連携・API 要件

| 外部サービス            | 概要     | 利用目的                                                              |
| ----------------------- | -------- | --------------------------------------------------------------------- |
| Google Maps Platform    | 地図表示 | 世界遺産の位置情報表示                                                |
| Firebase Storage        | 画像保存 | 訪問記録・日記用の写真保存                                            |
| Supabase（PostgreSQL）  | 業務 DB  | 世界遺産・訪問記録・日記などの業務データ管理（※バックエンド経由のみ） |
| Stripe                  | 決済処理 | プレミアム機能の課金処理                                              |
| Cloud Firestore         | 決済 DB  | Stripe 連携用の決済・課金状態管理                                     |
| Firebase Authentication | 認証     | ユーザー認証・ユーザー管理                                            |
| OpenAI API              | AI 処理  | 世界遺産解説生成・旅行プラン提案                                      |

## 10. 制約事項・前提条件

- 技術スタック

  - フロント：Next.js
  - バックエンド：FastAPI
  - DB：PostgreSQL（Supabase）

- MVP では以下を対象外とする

  - 周辺観光・宿泊提案
  - SNS 共有機能
  - 管理画面

- Supabase は開発用プロジェクトを使用

---

## 11. リスクと対応方針

| リスク               | 対応                                                         |
| -------------------- | ------------------------------------------------------------ |
| データ不整合         | 事前テストデータ投入                                         |
| 認証方式変更         | 事前周知・影響範囲確認                                       |
| 情報量過多           | 見どころ TOP3 に限定                                         |
| 遺産群の複雑性       | 代表地点方式を採用                                           |
| コンプリート後の離脱 | 将来的に対象世界遺産を日本から世界へ拡大、期間限定バッジ検討 |
